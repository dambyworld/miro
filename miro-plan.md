# Miro 전역 실행 구성 계획

## 목표

`target/release/miro`를 전역 명령처럼 실행할 수 있게 만드는 구성 작업을 계획하고 구현한다.

## 현재 상태

- 실행 파일은 현재 프로젝트 내부의 `target/release/miro`에 생성된다.
- 이 경로는 직접 입력해야 하므로 전역 명령처럼 쓰기 불편하다.
- 현재 사용자 환경의 `PATH`에는 `~/bin`이 포함되어 있다.
- 실제로 `rev-start`, `rev-done` 같은 전역 커맨드도 `~/bin`에 배치되어 있다.

## 문제 정의

현재 `miro`는 빌드 산출물 경로를 직접 알아야만 실행할 수 있다.  
전역 실행 구성을 하려면 아래 조건을 만족해야 한다.

1. 사용자가 어느 디렉터리에서든 `miro`만 입력해 실행할 수 있어야 한다.
2. 새 릴리스 빌드가 생겼을 때 전역 실행 경로도 일관되게 갱신 가능해야 한다.
3. 기존 `PATH` 구성과 충돌하지 않아야 한다.
4. 설치/업데이트/제거 절차가 단순해야 한다.

## 조사 결과

### 1. 현재 PATH

현재 쉘 `PATH`에는 아래 경로가 포함되어 있다.

- `~/bin`
- `~/.local/bin`
- `/opt/homebrew/bin`
- `/usr/local/bin`

이 중 사용자 권한으로 가장 안전하게 관리할 수 있는 경로는 `~/bin`이다.

### 2. 현재 전역 커맨드 패턴

`~/bin` 안에는 이미 아래와 같은 전역 스크립트가 있다.

- `rev-start`
- `rev-done`
- `rev-init`
- `claude-memory-sync`

즉, `miro`도 같은 패턴으로 `~/bin`에 두는 것이 현재 환경과 가장 자연스럽다.

## 구성 방식 후보

### 안 A. `~/bin/miro` 심볼릭 링크

구성:

- `~/bin/miro -> /Users/cozyai/dev/.miro/target/release/miro`

장점:

- 가장 단순하다.
- 현재 바이너리를 바로 가리키므로 구현량이 적다.
- `PATH`가 이미 잡혀 있어서 추가 환경 설정이 거의 없다.

단점:

- 바이너리 위치가 고정 경로에 묶인다.
- 빌드 산출물이 지워지면 링크가 깨진다.
- 다른 머신이나 다른 checkout 경로로 옮길 때 유연성이 떨어진다.

### 안 B. `~/bin/miro` 래퍼 스크립트

구성:

- `~/bin/miro` 스크립트가 `/Users/cozyai/dev/.miro/target/release/miro "$@"`를 실행

장점:

- 심볼릭 링크보다 오류 메시지나 fallback 처리를 넣기 쉽다.
- 바이너리가 없을 때 안내 문구를 제공할 수 있다.
- 추후 `cargo build --release` 자동 호출 여부 같은 정책을 넣기 쉽다.

단점:

- 심볼릭 링크보다 한 단계 복잡하다.
- 실제 실행 파일과 래퍼를 함께 관리해야 한다.

### 안 C. `cargo install --path .`

구성:

- cargo가 바이너리를 cargo 전역 설치 경로로 설치

장점:

- Rust 표준 배포 방식에 가깝다.
- cargo 생태계 관점에서 관리가 명확하다.

단점:

- 현재 로컬 수정 후 즉시 반영하는 흐름에는 다소 무겁다.
- 개발 중인 바이너리를 자주 갱신할 때는 재설치 비용이 든다.
- 이번 저장소 운영 방식에는 `~/bin` 직접 관리보다 덜 직관적일 수 있다.

## 권장안

현재 환경과 사용 패턴 기준 1순위는 `~/bin/miro` 래퍼 스크립트다.

권장 이유:

1. `~/bin`이 이미 `PATH`에 포함되어 있다.
2. 기존 전역 커맨드 관리 방식과 일관된다.
3. 바이너리 누락 시 친절한 오류 메시지를 넣을 수 있다.
4. 추후 설치 스크립트나 업데이트 정책으로 확장하기 쉽다.

2순위는 단순 심볼릭 링크다.  
빠르게 붙이기에는 좋지만, 운영성과 오류 대응은 래퍼 스크립트보다 약하다.

추가 원칙:

1. 전역 실행 구성은 `target/release/miro` 자체를 옮기는 방식보다, 프로젝트 안의 릴리스 바이너리를 호출하는 얇은 진입점으로 유지한다.
2. 수동으로 `~/bin/miro`를 편집하기보다, 저장소 안의 설치 스크립트가 이를 생성하거나 갱신하는 흐름을 기본안으로 둔다.
3. 전역 실행 구성은 개발 중인 저장소 경로를 기준으로 동작한다는 점을 문서에 명시한다.

## 권장 구현 방향

### 1단계. 전역 실행 방식 확정

- `~/bin/miro`를 사용할지 확인
- 심볼릭 링크 대신 래퍼 스크립트를 기본안으로 확정

### 2단계. 래퍼 스크립트 설계

래퍼는 아래 동작을 수행하도록 설계한다.

1. `target/release/miro` 존재 여부 확인
2. 없으면 빌드 필요 메시지 출력
3. 있으면 `"$@"`를 그대로 전달해 실행
4. 가능한 경우 저장소 루트 경로를 한 곳에서만 관리하도록 구성

권장 구현 형태:

- 저장소 내부에 설치 스크립트 예: `scripts/install-miro-global.sh`
- 이 스크립트가 `~/bin/miro`를 생성 또는 갱신
- `~/bin/miro`는 실제 바이너리 호출만 담당

예상 동작 형태:

```bash
#!/bin/zsh
exec /Users/cozyai/dev/.miro/target/release/miro "$@"
```

실제 구현 시에는 경로 존재 확인, 오류 메시지, 설치 대상 경로 확인을 추가할 수 있다.

### 3단계. 실행 권한 및 검증

- `~/bin/miro`에 실행 권한 부여
- `which miro` 결과 확인
- 임의 디렉터리에서 `miro list` 실행 확인
- `miro` 실행 시 현재 작업 디렉터리와 무관하게 동일한 바이너리가 호출되는지 확인

### 4단계. 업데이트 절차 정리

전역 실행 구성 후에는 업데이트 절차도 같이 정리해야 한다.

- 코드 수정
- `cargo build --release`
- 필요 시 설치 스크립트 재실행
- 전역 실행 파일은 동일한 진입점을 계속 사용

즉, 래퍼 방식이면 빌드 산출물만 갱신되면 된다.

### 5단계. 제거 절차 정리

- `~/bin/miro` 삭제 방식 정리
- 설치 스크립트를 썼다면 대응되는 제거 절차도 같이 정의
- 프로젝트 자체 삭제 전에 전역 진입점을 먼저 제거하도록 안내

문서 반영 내용:

- 전역 실행 확인 방법

- 제거 방법

## 검증 기준

아래 조건을 만족하면 구성 완료로 본다.

1. 어느 디렉터리에서든 `miro` 명령이 실행된다.
2. `miro list`가 정상 동작한다.
3. `miro`가 `target/release/miro` 최신 빌드를 사용한다.
4. 전역 실행 경로가 현재 `PATH`와 충돌하지 않는다.
5. 설치/업데이트/제거 절차가 문서화된다.
6. 저장소 경로가 바뀌었을 때 어떤 조치가 필요한지 사용자가 문서만 보고 이해할 수 있다.

## 리스크

1. 프로젝트 경로가 바뀌면 래퍼 내부 절대 경로도 같이 깨질 수 있다.
2. 릴리스 바이너리를 다시 빌드하지 않으면 전역 실행이 오래된 버전을 가리킬 수 있다.
3. `~/bin`이 일부 셸 세션에서 로드되지 않으면 명령을 찾지 못할 수 있다.
4. 수동 설치 방식만 남기면 나중에 재설치나 제거 절차가 일관되지 않을 수 있다.

## 리스크 대응

1. 래퍼에 바이너리 누락 시 명확한 오류 메시지를 넣는다.
2. 매뉴얼에 `cargo build --release`를 업데이트 절차로 명시한다.
3. `echo $PATH`와 `which miro`로 설치 직후 검증하도록 한다.
4. 설치와 제거를 가능한 한 동일한 스크립트 진입점으로 정리한다.

## 구현 결과

아래 방식으로 구현한다.

1. 저장소 내부에 `scripts/install-miro-global.sh`를 추가한다.
2. 이 스크립트가 `~/bin/miro` 래퍼를 생성, 갱신, 제거한다.
3. 래퍼는 `/Users/cozyai/dev/.miro/target/release/miro`를 호출한다.
4. 릴리스 바이너리가 없을 때는 `cargo build --release` 안내 메시지를 출력한다.

구현 후 검증 항목:

1. `cargo test`
2. `cargo build --release`
3. `scripts/install-miro-global.sh install`
4. `which miro`
5. 임의 디렉터리에서 `miro list`

검증 결과:

1. `cargo test` 통과
2. `cargo build --release` 통과
3. `scripts/install-miro-global.sh install`로 `/Users/cozyai/bin/miro` 설치 완료
4. `which miro` 결과 `/Users/cozyai/bin/miro`
5. `/tmp`에서 `miro list --provider codex` 정상 출력 확인
