# Miro 개발 계획

## 목표

`codex`, `claude-code` 세션을 터미널에서 통합 관리하는 TUI 프로그램 `Miro`의 구현 계획을 수립한다.

이번 단계는 계획 수립만 수행하며, 구현은 보류한다.

## 요구사항 정리

1. `codex`, `claude-code` 세션 목록 조회
2. 특정 세션 재진입
3. 특정 세션 삭제

## 문제 정의

`codex`와 `claude-code`는 각각 세션 저장 위치, 메타데이터 구조, 재진입 명령, 삭제 방식이 다를 가능성이 높다.  
`Miro`는 이 차이를 내부 어댑터로 흡수하고, 사용자에게는 일관된 TUI 경험을 제공해야 한다.

## 계획 범위

이번 계획은 아래 범위까지만 다룬다.

- 세션 탐색 방식 정의
- TUI 정보 구조 정의
- 재진입/삭제 동작 흐름 정의
- 구현 순서와 검증 기준 정의

이번 계획에서 제외한다.

- 실제 코드 작성
- 패키징/배포 자동화
- 세션 포맷 변경 또는 마이그레이션

## 핵심 가정

아래 가정은 구현 전 반드시 검증해야 한다.

1. `codex` 세션을 식별할 수 있는 로컬 저장소 또는 메타데이터 파일이 존재한다.
2. `claude-code` 세션도 동일하게 로컬에서 식별 가능하다.
3. `codex`, `claude-code` 모두 기존 세션으로 재진입 가능한 `resume` 커맨드를 제공한다.
4. 세션 삭제는 로컬 파일 삭제만으로 충분하거나, 각 CLI가 제공하는 안전한 삭제 명령이 있다.

## 조사 항목

구현 전 아래 항목을 먼저 확인한다.

### 1. 세션 저장 구조 조사

- `codex` 세션 디렉터리 위치
- `codex` 세션 메타데이터 형식
- `claude-code` 세션 디렉터리 위치
- `claude-code` 세션 메타데이터 형식
- 마지막 사용 시각, 작업 디렉터리, 세션 제목 추출 가능 여부

### 2. 재진입 방식 조사

- `codex`에서 특정 세션을 다시 여는 공식 명령 존재 여부
- `claude-code`에서 특정 세션을 다시 여는 공식 명령 존재 여부
- 세션 ID만으로 가능한지, 작업 디렉터리가 추가로 필요한지 확인
- 실패 시 오류 메시지 및 복구 전략 정의

### 3. 삭제 방식 조사

- 공식 삭제 명령 존재 여부
- 파일 직접 삭제가 허용되는지 여부
- 삭제 후 인덱스/캐시 정리 필요 여부
- 삭제 전 확인 프롬프트가 필요한지 여부

## 제품 구조 제안

### 1. 내부 도메인 모델

공통 세션 모델을 먼저 정의한다.

```text
SessionRecord
- provider: codex | claude-code
- session_id: string
- title: string | null
- cwd: string | null
- updated_at: datetime | null
- summary: string | null
- raw_ref: provider-specific reference
```

핵심 원칙은 "표시는 공통, 실행은 provider별"이다.

### 2. Provider Adapter 계층

각 도구별 차이를 흡수하기 위해 어댑터 패턴을 사용한다.

```text
SessionProvider
- list_sessions() -> list[SessionRecord]
- resume_session(session_id) -> ExecPlan
- delete_session(session_id) -> DeletePlan
- inspect_session(session_id) -> SessionRecord
```

예상 구현체:

- `CodexSessionProvider`
- `ClaudeCodeSessionProvider`

이 구조를 쓰면 이후 provider 추가가 쉬워진다.

### 3. TUI 구조

초기 화면은 단일 세션 목록 화면으로 시작한다.

- 상단: provider 필터, 검색어, 정렬 기준
- 본문: 세션 목록
- 하단: 키 바인딩 도움말

선택 상태 표현:

- 현재 커서가 위치한 세션은 하이라이트 배경, 포인트 컬러, 또는 강조된 텍스트 스타일로 즉시 구분되어야 한다.
- 선택 상태는 한눈에 들어와야 하며, 삭제 대상이나 재진입 대상이 무엇인지 혼동되지 않아야 한다.

권장 키 바인딩:

- `Up` / `Down`: 이동
- `/`: 검색
- `f`: provider 필터 전환
- `Enter`: 세션 재진입
- `d`: 삭제 확인
- `r`: 새로고침
- `q`: 종료

### 4. 동작 흐름

### 세션 목록 조회

1. 두 provider에서 세션을 읽는다.
2. 공통 모델로 정규화한다.
3. 최신 사용 시각 기준으로 정렬한다.
4. 목록에는 세션 ID보다 사람이 이해할 수 있는 제목 또는 마지막 대화 문구를 우선 표시한다.
5. 보조 정보로 provider, cwd, 최근 사용 시각을 함께 표시한다.

### 세션 재진입

1. 선택한 세션의 provider를 확인한다.
2. 해당 provider의 resume 실행 계획을 만든다.
3. 현재 TUI를 종료하거나 suspend 후 외부 명령으로 진입한다.
4. 진입 실패 시 TUI로 복귀해 오류를 표시한다.

### 세션 삭제

1. 삭제 확인 모달을 띄운다.
2. provider별 삭제 계획을 실행한다.
3. 성공 시 목록에서 제거한다.
4. 실패 시 삭제되지 않은 이유를 보여준다.

## 기술 방향 제안

구현 언어는 아직 확정하지 않았지만, 현재 로컬 환경을 기준으로 보면 `Rust + Ratatui`를 우선 검토하는 것이 적절하다.  
`rustc`, `rustup`은 이미 설치되어 있고, `go`는 아직 설치되어 있지 않다.  
따라서 `Go + Bubble Tea`는 기술 검토 이전에 Go 설치 비용이 추가된다.

### 안 A. Python + Textual

장점:

- TUI 생산성이 높다.
- 리스트, 모달, 비동기 갱신 구현이 빠르다.
- 로컬 파일 탐색과 subprocess 연동이 단순하다.

단점:

- 배포 시 Python 런타임 의존성이 생긴다.

### 안 B. Go + Bubble Tea

장점:

- 단일 바이너리 배포가 쉽다.
- CLI/TUI 툴로 적합하다.
- 파일 시스템, 프로세스 실행 처리에 강하다.

단점:

- 초기 UI 구현 속도는 Python 대비 느릴 수 있다.

### 안 C. Rust + Ratatui

장점:

- 단일 바이너리 배포에 유리하다.
- 성능과 안정성이 좋다.
- 장기적으로 유지보수성과 타입 안정성을 확보하기 좋다.

단점:

- 초기 구현 속도가 상대적으로 느릴 수 있다.
- subprocess, 파일 파싱, 상태 관리까지 포함하면 초반 진입 비용이 높다.

현재 목표가 "빠른 검증"이라면 `Python + Textual`,  
장기적으로 "배포성"과 "생산성 균형"을 원하면 `Go + Bubble Tea`,  
장기적으로 "안정성"과 "단일 바이너리 품질"을 더 중시하면 `Rust + Ratatui`가 적합하다.

## 기술 선택 승인 기준

구현 착수 전 언어와 TUI 프레임워크는 아래 기준으로 최종 승인한다.

1. `codex`, `claude-code` 세션 파싱을 무리 없이 처리할 수 있는가
2. 외부 CLI 재진입과 삭제 명령 실행을 단순하게 연결할 수 있는가
3. 개발 속도와 배포 편의성 중 현재 우선순위에 맞는가
4. 단일 바이너리 필요성이 실제 요구인지, 아니면 빠른 프로토타이핑이 더 중요한지 명확한가

승인 원칙:

- 1순위 후보는 `Rust + Ratatui`
- 2순위 후보는 `Go + Bubble Tea`

## Bubble Tea vs Ratatui 비교

두 후보는 모두 단일 바이너리 배포에 적합하지만, 현재 요구사항 기준으로는 아래 차이가 있다.

현재 로컬 환경 기준 비교 대상은 아래 두 가지다.

- `Ratatui`로 바로 개발하는 방식
- `Go`를 새로 설치한 뒤 `Bubble Tea`로 개발하는 방식

### Ratatui가 유리한 점

- 강한 타입 안정성과 소유권 모델 덕분에 상태 전이가 복잡해져도 구조를 엄격하게 유지하기 쉽다.
- 장기 유지보수성 측면에서 런타임 오류를 줄이기 좋다.
- 파일 파싱, 세션 모델 정규화, 삭제 안전성 같은 로직을 더 보수적으로 다루기에 적합하다.

### Bubble Tea가 유리한 점

- TUI 생산성이 높고 초기 화면 구현 속도가 빠르다.
- 외부 프로세스 실행과 파일 시스템 접근이 단순해서 빠른 실험에 유리하다.
- Rust보다 러닝커브가 완만해서 MVP를 더 빨리 만들 수 있다.

### 현재 로컬 환경 기준 차이

- `Rust + Ratatui`는 추가 설치 없이 바로 개발을 시작할 수 있다.
- `Go + Bubble Tea`는 먼저 Go 설치가 필요하므로 착수 비용이 더 크다.
- 현재 단계가 구현이 아니라 계획 수립이긴 하지만, 실제 착수 시점의 마찰 비용까지 고려하면 `Rust + Ratatui` 쪽이 더 자연스럽다.

### Miro 기준 판단

- `Miro`가 장기적으로 세션 메타데이터 파싱, provider 확장, 삭제 안전성까지 책임지는 툴이라면 `Rust + Ratatui`가 더 적합하다.
- 단기간에 TUI 동작과 resume/delete 흐름만 먼저 검증하려면 `Go + Bubble Tea`가 더 빠를 수 있다.
- 다만 현재는 Go가 설치되어 있지 않기 때문에, "지금 바로 시작할 수 있는가"까지 포함하면 `Rust + Ratatui`의 우위가 더 커진다.
- 현재 문서의 방향성은 "지속적으로 쓰는 세션 관리자"에 가깝기 때문에 기본 우선순위는 `Rust + Ratatui`로 둔다.

기본 권장안:

- 특별한 제약이 없으면 `Rust + Ratatui`를 우선 검토한다.
- 단, 이후 Go를 설치한 뒤 빠른 TUI 프로토타입이 더 중요하다고 판단되면 `Go + Bubble Tea`를 예외적으로 선택할 수 있다.

현재 문서 기준으로는 세션 저장 구조 조사가 끝난 뒤 기술 스택을 확정하되, 우선순위는 `Rust + Ratatui` 쪽에 둔다.

## 단계별 구현 계획

### 1단계. 세션 저장소 리서치

- `codex` 세션 저장 위치와 형식 확인
- `claude-code` 세션 저장 위치와 형식 확인
- 샘플 세션 3~5개 기준 필드 추출 가능성 확인
- 재진입 명령과 삭제 방식 확인

산출물:

- provider별 세션 구조 메모
- resume/delete 명령 예시

### 2단계. 공통 세션 모델 설계

- provider별 raw data를 `SessionRecord`로 매핑
- 제목/경로/최근 시각 fallback 규칙 정의
- 정렬/검색 대상 필드 정의

산출물:

- 모델 스펙 문서
- provider adapter 인터페이스

### 3단계. Provider Adapter 구현

- `CodexSessionProvider` 구현
- `ClaudeCodeSessionProvider` 구현
- 목록/단건 조회/삭제 실행 계획 구성

산출물:

- provider 단위 테스트
- 샘플 세션 파싱 검증

### 4단계. TUI 기본 화면 구현

- 목록 렌더링
- provider 필터
- 검색
- 새로고침
- 빈 상태 / 오류 상태 처리

산출물:

- 세션 브라우저 MVP

### 5단계. 재진입 연동

- 선택 세션 resume 실행
- TUI suspend/exit 전략 확정
- 실패 복귀 UX 정리

산출물:

- provider별 resume 검증 시나리오

### 6단계. 삭제 연동

- 삭제 확인 모달
- 삭제 후 목록 즉시 반영
- 실패 및 권한 오류 처리

산출물:

- 안전 삭제 UX 완성

### 7단계. 마감 작업

- 로그/디버그 옵션
- 도움말 문구 정리
- README 초안
- 패키징 방향 정리

## UX 원칙

- provider 차이는 내부 구현에서만 처리하고 화면 경험은 통일한다.
- 삭제는 항상 명시적 확인을 거친다.
- 재진입은 가장 빠른 단축키 동선으로 제공한다.
- 세션이 많아도 검색과 정렬이 즉시 동작해야 한다.
- 세션 메타데이터가 일부 비어 있어도 목록이 깨지지 않아야 한다.
- 목록에서 현재 선택된 세션은 하이라이트나 텍스트 색상 변화로 분명하게 강조되어야 한다.

## 리스크

1. 두 CLI가 공식적인 세션 재진입 인자를 제공하지 않을 수 있다.
2. 세션 파일 포맷이 비공개 또는 버전별로 바뀔 수 있다.
3. 삭제가 단순 파일 삭제로 끝나지 않을 수 있다.
4. TUI 내부에서 외부 세션으로 자연스럽게 handoff하는 UX가 어색할 수 있다.

## 리스크 대응

1. 공식 명령이 없으면 "실행 계획 출력" 또는 "복사 가능한 명령 제공"으로 1차 우회한다.
2. provider 파서를 느슨하게 설계하고 raw metadata fallback을 둔다.
3. 삭제는 기본적으로 soft-fail 처리하고 명확한 오류 원인을 보여준다.
4. 초기 버전은 TUI 종료 후 재진입하는 방식으로 단순화한다.

## 검증 기준

아래 조건을 만족하면 MVP 기준 충족으로 본다.

1. `codex`, `claude-code` 세션이 하나의 목록으로 보인다.
2. 최근 사용 시각 기준 정렬이 동작한다.
3. `codex`, `claude-code`의 `resume` 커맨드를 통해 특정 세션 재진입이 provider별로 성공한다.
4. 특정 세션 삭제 후 목록에서 즉시 사라진다.
5. 잘못된 세션이나 손상된 메타데이터가 있어도 프로그램이 죽지 않는다.

## 권장 초기 산출물

구현 착수 시 우선순위는 아래 순서가 적절하다.

1. 세션 저장 위치/포맷 조사 문서
2. 공통 모델과 provider adapter 인터페이스
3. 목록 조회 전용 프로토타입
4. 재진입 기능
5. 삭제 기능

현재 상태에서는 구현보다 먼저 `codex`, `claude-code`의 실제 세션 저장 구조와 재진입/삭제 진입점을 확인하는 것이 우선이다.  
따라서 이번 작업은 `Miro` 구현 계획 문서 작성까지로 종료한다.
